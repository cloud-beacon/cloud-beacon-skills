# Cloud Beacon Voice & Tone

Cloud Beacon is a trusted Dynamics 365 Finance & Operations consulting partner. Writing should reflect the same qualities we bring to engagements: **precision, clarity, confident expertise.**

## Core voice attributes

1. **Professional & authoritative** — Earned confidence. Our consultants are experts; the writing should reflect that without arrogance.
2. **Clear & direct** — Lead with the conclusion, then support with detail. Never bury the key point. Readers are busy decision-makers.
3. **Technically precise** — Name specific technologies, features, and standards by their correct names. "Trade Agreement journals with price groups," not "standard pricing feature." Precision builds credibility.
4. **Approachable, not casual** — Consultants, not academics. Write for humans. Use contractions sparingly. Skip jargon that doesn't add meaning. If a simpler word works, use it.
5. **Action-oriented** — Every document makes clear what happens next. Recommendations include concrete steps. Proposals include specific deliverables. Reports include actionable findings.

## Tone by document type

| Document type        | Tone                                  | Example phrasing                                          |
| -------------------- | ------------------------------------- | --------------------------------------------------------- |
| Proposal / SOW       | Confident, structured, contractual    | "Cloud Beacon will deliver..."                            |
| Technical document   | Precise, instructional, thorough      | "The integration uses the Claim-Check pattern to..."      |
| Email / memo         | Warm, concise, action-oriented        | "Here's what we recommend and why..."                     |
| Presentation         | Bold, visual, high-impact             | "3 risks. 2 recommendations. 1 path forward."             |
| Report / review      | Objective, evidence-based, constructive | "Finding: The batch job lacks idempotency checks..."    |

## Structural rules

- **Lead with impact.** Executive summaries first. Most important information opens the document.
- **Progressive disclosure.** High-level overview, then detailed breakdown for those who need it.
- **Number sections** in documents longer than two pages. Aids navigation and meeting references.
- **Short paragraphs.** Two to four sentences max. Dense paragraphs signal unedited thinking.

## Language rules

- **Active voice.** "Cloud Beacon will configure the system," not "The system will be configured by Cloud Beacon."
- **Present tense for capabilities.** "This integration handles 10,000 records per hour," not "will handle."
- **Specific over vague.** "Reduces processing time by 40%," not "Significantly improves performance."
- **Cut filler.** Remove "In order to" → "To." "It should be noted that" → state the thing. "At this point in time" → "Now."
- **Consistent terminology.** Pick one term and stick with it. Don't alternate between "D365 F&O," "Dynamics 365," and "the system" in one document.

## Formatting conventions

- **Bold** for key terms on first use and for emphasis in recommendations.
- *Italics* for document titles and defined terms.
- `Code font` for technical identifiers, field names, table names, API endpoints.
- Bullet lists for three or more parallel items; numbered lists for sequential steps.

## Internal → customer-facing transformation

When converting internal documents to customer-facing versions:

1. **Remove internal jargon.** Replace team-specific shorthand with full terms.
2. **Add context.** Internal docs assume shared knowledge; external docs must stand alone.
3. **Elevate the opening.** Replace casual openings with professional executive summaries.
4. **Structure for scanning.** Add headings, break long paragraphs, use tables for comparisons.
5. **End with next steps.** Every customer-facing document closes with clear action items.
6. **Check audience.** Match technical depth to reader (executive vs. technical lead vs. developer).

## Hard constraints

- **Never alter technical facts, numbers, or specific claims** during voice/tone transformation. Voice changes only — content integrity is non-negotiable.
- **Never invent specifics.** If the source says "improves throughput," don't write "improves throughput by 40%."
- **Preserve all named entities** — client names, product names, version numbers, dates — exactly as written.
